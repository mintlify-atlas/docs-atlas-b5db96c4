## 2.1.0 (2022-05-09)

- Support for using sdkgen in Serverless services (such as Google Cloud Functions or AWS Lambda), or alongside existing Node.js servers, like Express.js. To do this, simply create a `SdkgenHttpServer` instance as usual, and then register the `handleRequest` property as a function to receive HTTP requests. See details in the [documentation](https://sdkgen.github.io/dev/typescript_nodeserver/intro/#uso-do-sdkgen-com-um-servidor-http-existente-ou-cloud-functions). See [#678](https://github.com/sdkgen/sdkgen/pull/678).
- Added `ctx.response.headers` and `ctx.response.statusCode` to allow modifying the headers and status code of the response of any request, especially useful for REST requests. See [#676](https://github.com/sdkgen/sdkgen/pull/676).
- Fixes in type validation when more than one type has the same name in different locations. See [#672](https://github.com/sdkgen/sdkgen/pull/672).
- Second argument of `SdkgenHttpServer` is now optional when `ExtraContext` is not used, simplifying code when dependency injection is not necessary.
- Added support for ES2022 with TypeScript. See [#682](https://github.com/sdkgen/sdkgen/pull/682).
- Improvement in the playground's "About" dialog. See [#683](https://github.com/sdkgen/sdkgen/pull/683).
- Dependency updates.

## 2.0.0 (2022-02-07)

We've arrived at sdkgen 2.0.0, after almost 1 year of development, bringing many improvements and new features, with few changes that may impact projects. The expectation is that the update can be applied in most projects without any code changes. It is always recommended that the server updates first, before the clients. Let's see what we have in terms of new features:

### Flutter 2 Support (sound null safety and Flutter Web)

Flutter 2 was released alongside Dart 1.12 and brought optional nullable types support, a significant evolution. A good part of the ecosystem has already migrated and sdkgen is now built the same way. Both the generated Dart code and the `sdkgen_runtime` package use nullable types. In addition, Flutter Web is supported, just like Flutter for Android and iOS, offering the same functionalities.

One detail that makes using this version easier is that previously the SDK had to be initialized inside a widget by passing access to the `BuildContext`. This is no longer necessary and can be used anywhere in the application, regardless of the UI.

### Native iOS Client (experimental)

A new native iOS client has been added, along with the Swift code generator. It can be used with the `swift_ios` or `rxswift_ios` target in the generator. In this version, this target is already functional and supports all features, however it has not been tested enough in production yet. In case of problems, please report. With sufficient testing happening, we can mark it as stable in one of the next versions.

### F# Server (experimental)

The original support for running sdkgen on .NET was built based on .NET Core using C#. This entire base has been rebuilt to make room for a new version in F#, much more modern and making it possible to be used in both F# and C# projects. Method implementation is easy and straightforward, just like in Node.js, maintaining the same simplicity. Currently the base features of sdkgen are supported, but some optional ones are not supported by this target, such as REST support and compatibility with legacy versions. It is being released as experimental as it has not been tested enough in production yet. The expectation is to release it as a stable and complete version in the near future.

### New Playground

Sdkgen has had an embedded playground since version 1.0.0, making it possible to inspect an API, get to know the available functions and invoke them with data for testing. We are now including a new playground, with support for:

- Explore existing functions, along with documentation of what they do and the types they receive or return.
- See how the function should be called in the languages supported by sdkgen.
- Call functions, writing the arguments you want to pass.
- Generate and download the generated code in all supported targets without having to install anything.
- Use the advanced editor, where it is possible to write logic in JavaScript calling one or more functions to run directly in your browser.

The new playground can even be used with APIs that have not yet been updated to sdkgen 2.0.0.

Access at: [https://sdkgen.github.io/playground/](https://sdkgen.github.io/playground/) or on the `/playground` route of your API. `introspection` must be enabled.

### Type Unions (experimental)

In many cases it is necessary for a property to receive more than one data type depending on the situation, for example when a list of events is returned. Until now there was no good way to represent these cases, but now sdkgen supports aggregating additional data in the enum. An example of how this can be used:

```sdkgen
type Event enum {
  info(msg: string)
  transaction(id: uuid, value: money, date: datetime, description: string)
  refund(id: uuid)
  ping
}

fn getEvents(): Event[]
```

In this example an `Event` can be any of the 4 possible types, depending on which one it is, it will be accompanied by some additional data. With this it is possible to express much more complex API scenarios, without having to make workarounds.

Currently this model works in Node.js and on the Web with TypeScript, but is not yet enabled in other targets, which is why it is marked as experimental. We hope to provide it in all languages in the future.

### Recursive Types

Another difficult case to express is when types need to be arbitrarily recursive. For example, observe the case of comments on a post, where a comment can have other comments inside. It is now possible to declare it like this:

```sdkgen
type Post {
  body: string
  authorId: uuid
  comments: Comment[]
}

type Comment {
  body: string
  authorId: uuid
  comments: Comment[]
}
```

In this case comments can contain comments as many times as necessary, using an empty array to represent a comment that has no other comments inside. Recursive types can be used together with type unions as well.

We hope with this to increase the expressiveness of the API description, representing real scenarios and avoiding the need to use the `json` type.

### Other Changes

All features marked as deprecated in previous versions have been removed. If they were used in your project, you will need to update.

In addition, several internal fixes and improvements were applied, making the project more stable as a whole.

## 1.5.0 (2021-01-26)

### Middlewares

It is now possible to add multiple middlewares in the configuration of an API, making it possible to take actions depending on the arguments or the return.

```typescript
api.use(async (ctx, next) => {
  // Do anything with `ctx` here.
  // The function name is `ctx.request.name` and the arguments `ctx.request.args`.
  const reply = await next();
  // Do anything with the response here.
  return reply;
});
```

The previous hooks of `onRequestStart` and `onRequestEnd` continue to work, but are now deprecated.

You can, for example:

- Save request and response data in a log;
- Measure request statistics;
- Control rate limit;
- Block or redirect calls depending on the function and arguments;
- Implement cache;
- Implement retry depending on some specific error (calling `next` more than once);
- Modify the returned error (`try`/`catch` around `next`).

See [#161](https://github.com/sdkgen/sdkgen/pull/161).

### Additional Data in Errors ([#133](https://github.com/sdkgen/sdkgen/pull/133))

In sdkgen it has always been possible to define error types:

```sdkgen
error InvalidArgument
```

It is now possible to include additional data (of any type) along with an error:

```sdkgen
error InvalidArgument {
  argumentName: string
  reason: string
}
```

And the error can be thrown as:

```typescript
throw new InvalidArgument("Invalid argument!", { argumentName: "amount", reason: "Must be positive" });
```

Any type can be used, such as:

```sdkgen
error NoFunds uint?
```
