## Introduction

Sdkgen automatically generates type-safe API clients for multiple platforms. Each client provides:

- **Type safety**: Full TypeScript/language-specific types for all API calls
- **Automatic encoding/decoding**: Handles data serialization transparently
- **Error handling**: Structured error types with proper error propagation
- **Device tracking**: Automatic device information collection
- **Request hooks**: Customizable success and error callbacks

## Core Concepts

### SdkgenHttpClient

All platform clients are based on the `SdkgenHttpClient` class, which provides:

- **baseUrl**: The API endpoint URL
- **astJson**: The API schema definition (generated from your .sdkgen file)
- **errClasses**: Error class mappings for proper error handling
- **extra**: A Map/dictionary for passing additional metadata with requests

### Making Requests

Clients expose a `makeRequest` method that:

1. Encodes function arguments according to the API schema
2. Sends the request with device information
3. Decodes the response
4. Handles errors and transforms them into typed exceptions

## Error Handling

All clients use a structured error system:

### Base Error Types

- **SdkgenError**: Base error class with `message` and `type` properties
- **SdkgenErrorWithData**: Error class that includes additional typed data
- **Fatal**: Catch-all error for connection issues and unexpected failures

### Error Flow

1. Server returns an error response
2. Client matches error type to registered error class
3. Error is decoded with proper type information
4. Typed exception is thrown

```typescript
try {
  const result = await api.someFunction();
} catch (error) {
  if (error instanceof ValidationError) {
    console.error('Validation failed:', error.data);
  } else if (error instanceof Fatal) {
    console.error('Connection error:', error.message);
  }
}
```

## Extra Metadata

You can attach additional data to all requests using the `extra` property:

<Tabs>
  <Tab title="TypeScript">
    ```typescript
    client.extra.set('authToken', 'bearer-token-123');
    client.extra.set('sessionId', 'session-456');
    
    // All subsequent requests will include this metadata
    await client.makeRequest('getUser', { id: '123' });
    ```
  </Tab>
  
  <Tab title="Dart">
    ```dart
    client.extra['authToken'] = 'bearer-token-123';
    client.extra['sessionId'] = 'session-456';
    
    // All subsequent requests will include this metadata
    await client.makeRequest('getUser', {'id': '123'});
    ```
  </Tab>
</Tabs>

## Request Lifecycle

Every API call follows this flow:

<Steps>
  <Step title="Encode Arguments">
    Function arguments are encoded according to the API schema's type definitions.
  </Step>
  
  <Step title="Build Request">
    Request object is constructed with:
    - Function name
    - Encoded arguments
    - Device information
    - Extra metadata
    - Unique request ID
  </Step>
  
  <Step title="Send Request">
    HTTP POST request is sent to `{baseUrl}/{functionName}` with `Content-Type: application/sdkgen`.
  </Step>
  
  <Step title="Process Response">
    Response is parsed, decoded according to the return type schema, and returned.
  </Step>
  
  <Step title="Handle Errors">
    If an error occurs, it's transformed into the appropriate error class and thrown.
  </Step>
</Steps>
